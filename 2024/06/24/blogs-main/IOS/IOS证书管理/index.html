IOS证书管理

发表于2024-06-24|更新于2024-06-24|IOS
|阅读量:

苹果的目的

IOS的证书是用来确保一个目的，“如何保证用户手机安装的应用是由在苹果后台登记身份的开发者所开发的应用”，以确保用户的隐私安全。
那么，就可以先思考，如何做才能做到这个目的？

如何确保应用的安全，没有被篡改？

苹果所有的应用id都统一在苹果后台登记，那么在开发应用的时候，根据应用的唯一标识(bundleId)，去苹果后台申请一个证书，打包应用的时候把证书打包进应用，并用证书做数字签名。
然后手机在安装应用的时候，先从应用内取出证书，然后去苹果后台验证该证书是否合法，合法的话再通过证书做验签，就可以知道应用是否被篡改。

这样一看，Android的安全性确实差一点

IOS 证书可以分为两种：

  • Development：开发证书，用来开发和调试 App。一般证书名是 iPhone Developer: xxx。如果是多人协作的开发者账号，任意成员都可以申请自己的 Development 证书。
  • Distribution：发布证书，用来发布 App。一般证书名是 iPhone Distribution: xxx。只有管理员以上身份的开发者账号才可以申请，因此可以控制提交权限的范围。

沙盒权限限制

于Android不同，Android的权限申请都是在本地代码中进行配置，而IOS的应用权限申请也是要在苹果后台登记。
在创建AppId的时候，需要勾选对应的[Capabilities]能力，而在开发阶段，本地描述权限的文件是Entitlements文件，代码签名时，会将 Entitlements 文件（如有）嵌入二进制代码中，作为被签名内容的一部分，由代码签名保证其不可篡改性。
然后在安装或者发布的时候，苹果可以根据AppId来验证应用是否只用到了其所申请的权限。

安装设备的限制

苹果根据供应配置文件（Provisioning Profile）来控制设备是否可以安装某一个应用。
一个 Provisioning Profile 对应一个 Explicit ID 或 Wildcard ID。在苹果后台上手动创建一个 Provisioning Profile 时，需依次指定上述三项内容：

  • App ID：单选（沙盒权限，可多选）
  • 证书：Certificates，可多选，对应多个开发者
  • 设备：Devices，可多选，对应更多个开发设备

生成后的文件为一个 .mobileprovision的文件,Xcode编译的时候需要指定一个该文件用于正常编译。
在编译过程中会做一定的校验，其流程如下：

  1. 校验证书和沙盒权限是否于Provisioning Profile 相匹配，否则编译报错
  2. 校验Provisioning Profile内的证书文件是否已安装在本地，能够匹配对应的 Public/Private Key Pair，否则编译会报错。
  3. 使用证书的私钥进行签名
  4. 将签名、Provisioning Profile、应用程序打包到.ipa 中

苹果应用安装的验证和运行

测试包

  1. 系统根据查询苹果后台，校验应用的bundleId，沙盒权限，证书是否于后台记录的AppId等一致，否则无法启动 App。
  2. 使用本地内置的 CA 公钥对 Provisioning Profile 中匹配的 certificate 进行签名验证，从而确认匹配到的证书的合法性。
  3. 使用 Provisioning Profile 中的匹配的，且经过 CA 验证过的 certificate（即打包应用程序的开发者的证书）中取出公钥，对 App 进行签名验证，否则无法启动 App。
  4. 最后，设备系统会将设备的 Device ID 与 Provisioning Profile 中的 Devices 的 Device ID 进行匹配，否则无法启动 App。

    测试包的各种限制，感觉是为了限制普通开发者直接分发应用给消费者，损害App Store的利益

正式包

  1. 把验证过程交给了 App Store，App Store 验证通过后重新进行一次签名.
  2. 安装的时候，用设备上的 CA 公钥对 ipa 进行签名验证。
