Gradle_插件

Gradle本身只是提供了一些基本的概念和整体核心的框架，但是真正使用场景逻辑一般都是以插件扩展的方式来实现，比如Java 插件，Android Gradle插件，这些插件才是真正实现了构建Java项目或者Android项目的构建流程。

Java插件结构

Java插件会做一些默认设置和约定,如默认情况下

  • src/main/java ： 为源码存放路径
  • src/main/resources : 为需要打包的资源路径
  • src/test ： 为单元测试路径
    这些都是Java插件内置的两个源码路径，而通过Gradle，可以自行添加其他的路径 而对于默认的路径，也可以自行修改对应的路径

Android Gradle插件

AGP即Android gradle plugin ，实际就是一个Gradle的第三方插件，它是由Google的Android团队开发的。

Android Gradle插件分类

AGP根据项目可以分为三种类型， App应用工程，Library库工程，Test测试工程，它们对应的插件id分别为:
com.android.application
com.android.library
com.android.test

Android Gradle插件应用

因为AGP对于Gradle来说属于第三方插件，所以在应用插件的时候要先设置它的classpath，在实际Android开发中，就是在根项目里的build.gradle进行配置

然后在实际的App项目中就可以应用插件了

android{}是AGP提供的一个扩展类型，可以让开发者自由设置对应的版本。

Android Gradle插件结构


Android结构与Java类似，只不过一般Android都是用app子工程作为项目入口

  • proguard-rules.pro ： 混淆规则
  • .gitignore : git的忽略文件
  • gradle.properties : gradle配置信息文件
  • local.properties : 本地gradle配置信息
  • AndroidManifest.xml : 描述App的配置文件
  • res : 描述App的资源文件
    而Android的配置主要在app工程里面的build.gradle AGP工程的配置，都在android{}中，这是唯一的一个入口。通过它可以对AGP工程进行自定义的配置，其具体的实现是com.android.build.gradle.AppExtension，是Project的一个扩展。

Android Gradle常用DSL配置

defaultConfig

defaultConfig是一个默认配置，它是一个ProductFlavor。可以自定义不同的ProductFlavor，但如果没有特殊定义，那么默认的productFlavor就会使用这个，而通常在实际开发中，一般都是用默认配置。
defaultConfig的常用配置属性：

属性名 描述 类型
applicationId 应用id/包名，一般是唯一 String
minSdk 指定app支持的最低Android系统版本 Integer
targetSdk 指定app基于哪个Android Sdk开发 Integer
versionCode 应用版本号，应用覆盖安装的判断 Integer
versionName 应用版本名，用户看到的版本名 String
testInstrumentationRunner 单元测试使用的Runner，可以修改为自己自定义的Runner String
signingConfig 配置签名配置，但一般不会在这里进行配置，而会在 buildTypes 中进行配置 SigningConfig
proguardFile 配置混淆规则文件，但我们一般不会在这里使用该配置，而会在 buildTypes 中结合 minifyEnabled 一起使用 File
consumerProguardFile 这个属性只作用于我们创建的 library 中，包括我们以aar形式导入的 library ，或是直接创建的 library。它的作用是，负责该 library 被进行编译时的混淆规则，我们在 主App 的模块下则可以不用再管理各个 library 的混淆规则，会直接使用各个 library 的混淆规则文件 File
applicationSuffix 追加在applicationId字符串后面，形成最终的包名 String
flavorDimensions 风味维度 List
javaCompileOptions 配置java编译选项 JavaCompileOptions
multiDexEnabled 是否开启分包，超过65536个方法数就要开启，通常在buildType配置 Boolean
multiDexKeepFile 将需要的类打包进主包 File
multiDexKeepProguard 将需要的类打包进主包，与multiDexKeepFile作用一样，区别是写法上不同，一个是利用文件名，一个是类似混淆的匹配规则 String
ndk 内部有一个abiFilter属性，可以用于过滤abi，即编译打包的Apk只包含过滤后的abi NdkOptions
buildConfigField 可以在BuildConfig类中添加对应的字段和值，设置之后在编译完类文件就会生成对应的字段，注意设置字符串值的时候，必须加上双引号 String
resConfig 配置编译后保留的对应的res资源 List
resValue 对value资源进行添加，编译后可以通过R.string.x的方法获取 Map<String, String>

buildType

buildType和Java插件中的SourceSets类似，就是定义各种构建类型的域对象，而一般常用的就是debug，release等类型。
其中，buildType的配置有许多与defaultConfig一致:

属性名 描述 类型
applicationId 应用id/包名，一般是唯一 String
signingConfig 配置签名配置 SigningConfig
proguardFile 配置混淆规则文件 File
proguardFiles 配置混淆规则文件列表 List
consumerProguardFile 这个属性只作用于我们创建的 library 中，包括我们以aar形式导入的 library ，或是直接创建的 library。它的作用是，负责该 library 被进行编译时的混淆规则，我们在 主App 的模块下则可以不用再管理各个 library 的混淆规则，会直接使用各个 library 的混淆规则文件 File
consumerProguardFiles 这个属性只作用于我们创建的 library 中，包括我们以aar形式导入的 library ，或是直接创建的 library。它的作用是，负责该 library 被进行编译时的混淆规则，我们在 主App 的模块下则可以不用再管理各个 library 的混淆规则，会直接使用各个 library 的混淆规则文件 List
applicationSuffix 追加在applicationId字符串后面，形成最终的包名 String
javaCompileOptions 配置java编译选项 JavaCompileOptions
multiDexEnabled 是否开启分包，超过65536个方法数就要开启 Boolean
multiDexKeepFile 将需要的类打包进主包 File
multiDexKeepProguard 将需要的类打包进主包，与multiDexKeepFile作用一样，区别是写法上不同，一个是利用文件名，一个是类似混淆的匹配规则 String
buildConfigField 可以在BuildConfig类中添加对应的字段和值，设置之后在编译完类文件就会生成对应的字段，注意设置字符串值的时候，必须加上双引号 String
shrinkResources 是否启用未使用资源的收缩。默认值为false Boolean
debuggable 这个构建类型是否应该生成可调试的apk。 Boolean
testCoverageEnabled 是否为此构建类型启用了测试覆盖率。 Boolean
zipAlignEnabled 资源对齐，可以减少引用再运行时的内存消耗，release类型默认为true Boolean
manifestPlaceholders 占位符 Map<String, String>

signingConfig

一个App只有在签名之后才能被发布到应用市场，安装，使用，签名是保护App的方式，标记这个App的唯一性。签名被修改，就无法与之前的应用进行升级安装，确保了在应用升级过程中数据的安全性。
对App签名，需要有一个签名证书.

根据buildType，一般只有release和debug两种构建类型，可以针对这两种模式采用不同的签名方式。

  • debug类型
    一般对于debug类型，Android Sdk已经默认提供了一个debug模式的签名证书，所以在开发阶段不用处理证书就能进行安装。
  • release类型
    在发布的时候，就要对release包进行签名配置，使用自己的签名，确保应用的安全性。而配置签名就可以使用signingConfig，它是android{}的一个属性，可以进行设置 配置签名之后，要是签名生效还需要配置到对应的productFlavor上面，有两种方式，一种的配置在defaultConfig，另外一种是在buildType上面配置 关于签名的更多内容，可以参考以下文章

    一篇文章看明白 Android v1 & v2 签名机制
