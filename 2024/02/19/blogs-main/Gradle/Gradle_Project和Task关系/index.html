Project和Task

一个工程为一个Project，而Task是一个操作，一个原子操作，比如编译打包jar，上传jar到Maven等任务。一个Project里面可以有许多个Task任务。
Gradle就是通过每个Project下面各种各样的Task来完成对工程管理。

Gradle构建的生命周期


每个阶段的一些细节可以参考此地址
由于Gradle本质是通过各种Task来进行功能处理，那么如何管理和设置这些Task就是一个重要的问题。
从上面的图片可以看出，Gradle的构建主要分为3个阶段：

  • 初始化阶段 : 读取settring生成项目结构树。
  • 配置阶段 : 根据项目结构树生成对应的Projects，读取build.gradle文件中的task，确定每个task的依赖关系和执行顺序。
  • 执行阶段 : 真正执行对应的task.

Task访问

在Task创建的时候得知，任务的创建最终都是通过TaskContainer创建并保存在内部，所以访问task的时候也可以通过TaskContainer进行访问,如：tasks[“myTask”],其实际是会调用findByName(String name)实现.
另外，任务的访问可以分为两种，一种是通过路径访问，另外一种是通过方法名访问

Task内部的Action执行顺序

每一个Task实力内部会有一个actions集合对象，它是一个ArrayList,再调用doFirst/doLast的时候，实际就是往这个集合的头部或者尾部进行acion添加，而在执行Task的时候就会根据actions集合顺序进行执行

Task执行顺序

  • A任务依赖B任务 (即执行A任务之前要先执行B任务)

    同时dependsOn后面可以跟多个任务，比如： dependsOn myTask,myTask2,myTask3

  • A任务应该在B任务后执行
    即A任务和B任务同时执行话，A任务建议是在B任务之后执行，不是一种强依赖，也只是一种建议
    myTask.shouldRunAfter(myTask2)

  • A任务必须在B任务后执行
    即A任务和B任务同时执行话，A任务必须是在B任务之后执行，不是一种强依赖
    myTask.mustRunAfter(myTask2)

上面两种任务顺序和dependsOn的区别，个人理解是dependsOn主要是指任务执行的时候必须先执行某个依赖任务，而shouldRunAfter和mustRunAfter的作用是当两个任务要执行的时候，要有个先后顺序，但单个任务可以自行运行

Task启用和断言

Task中有一个enabled属性，通过设置true/false可以控制任务的启用和禁用，被禁用的任务在运行过程会跳过

Task中可以设置一个onlyIf，接受一个闭包，本质和enabled作用一致，只不过现在是通过条件来判断
