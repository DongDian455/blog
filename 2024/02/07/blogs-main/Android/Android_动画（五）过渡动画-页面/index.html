Android_动画（五）过渡动画-页面

发表于2024-02-07|更新于2024-02-07|Android
|阅读量:

页面之间的过渡动画

在Android_动画（四）过渡动画-布局文章中，主要了解了TransitionManager框架的执行过程，
并且举例了布局之间切换的一个效果。

而在页面之间切换的效果中，使用TransitionManager的同时，还要处理更多复杂的逻辑。因为页面之间切换会存在以下要素

  • 整体页面的切换效果
  • 共享元素的切换效果
  • 整体页面与共享元素分开执行的时候显示遮挡问题(GhostView & OverlayView)
  • 页面之间跨组件的通信(数据传递,Binder)

这里会存在两个Transition，通常是合并到一个TransitionSet进行执行。

一个页面切换的实现流程

  1. 开启页面转场效果
  2. 设置过渡动画并启动

页面切换涉及的相关类及方法

ActivityOptions

Activity切换的过渡动画信息，使用的是ActivityOptions对象，ActivityOptions类中提供了以下静态方法来设置Activity的转场动画，这些方法会返回一个ActivityOptions对象。

方法名 说明
makeClipRevealAnimation(View source, int startX, int startY, int width, int height) 以一个点为中心开始揭露
makeCustomAnimation(Context context, int enterResId, int exitResId) 类似 overridePendingTransition
makeThumbnailScaleUpAnimation(View source, Bitmap thumbnail, int startX, int startY) 放大一个 Bitmap 来过渡
makeScaleUpAnimation(View source, int startX, int startY, int width, int height) 以一个点为中心开始放大
makeSceneTransitionAnimation(Activity activity) 无共享元素过渡动画
makeSceneTransitionAnimation(Activity activity, View sharedElement, String sharedElementName) 单个共享元素的过渡动画
makeSceneTransitionAnimation(Activity activity, Pair<View, String>… sharedElements) 多个共享元素的过渡动画

动画设置的方法

整体页面效果

每个页面可以设置4种场景的页面动画

方法名 说明
void setEnterTransition(Transition transition) 进入当前Activity时候的展示动画,如：
ActivityA->ActivityB
B展示的时候显示的动画
void setReturnTransition(Transition transition) 关闭当前Activity时候展示的动画,如：
ActivityA.finish()
A关闭的时候显示的动画
void setReenterTransition(Transition transition) 当前Activity回到前台的时候展示的动画， 如：
ActivityA->ActivityB->ActivityA
从B回到A的时候,A展示的动画
void setExitTransition(Transition transition) 当前Activity退到后台的时候展示的动画，如:
ActivityA->ActivityB
A展示的动画

上述的4个场景对于Enter和Return，提供了2个开关

这两个开关的标志意义是 如果为true，表示两个页面的场景动画可以进行，为false则需要等待一方完成另外一方才开始，默认为true

需要特别注意！ 如果设置为true,可能会存在动画无法执行完毕的场景。
如：
在ActivityA->ActivityB中，在B中onCreate设置setAllowEnterTransitionOverlap（true）, 则可能会让A的exit动画无法执行完毕，因为ActivityB中的enter会同时进行，ActivityB会进入到前台，此时ActivityA页面会触发生命周期的onStop，而在onStop中Activity会取消所有Transition动画

共享元素效果

设置效果与页面的类似，只是针对的是共享元素

过渡动画延迟

在进场动画的一些场景，可能需要等待某些View准备好之后才进行，这个时候就需要使用两个方法

OverlayView和ViewGroupOverla

OverlayView和ViewGroupOverlay是Activity共享元素动画实现中比较重要的一环，主要处理非共享动画与共享动画分离执行的问题。

OverlayView是针对View的一个顶层附加层(即遮罩层)，它在View的所有内容绘制完成之后 再绘制
ViewGroupOverlay是针对ViewGroup的,是OverlayView的子类，它在ViewGroup的所有内容(包括所有的children)绘制完成之后 再绘制

特别的，Window提供了一些api来判断是否启用Overlay效果，如果不启用，则在原来的图层关系中进行动画，对于执行动画的元素，可能会出现被遮挡

GhostView

GhostView的在动画中的作用与OverlayView和ViewGroupOverla一样，他们是组合一起使用以达到动画分离执行的目的。

它的作用是在不改变view的parent的情况下，将view绘制在另一个parent下。
如果把View 添加到GhostView里，则默认会调用view的setTransitionVisibility方法 将view设置成在parent中不可见, 在GhostView里可见；调用GhostView的setVisibility方法设置 要么在GhostView中可见，要么在parent中可见

ExitTransitionCoordinator、EnterTransitionCoordinator

这两个类的作用是负责进场和退场两个组件之间的通信以及整体执行流程，两个类之间会相互持有对方，并通过Binder的方式进行通信。同时包含一些其他功能的封装（如保存view信息），旧activity使用ExitTransitionCoordinator，新activity使用EnterTransitionCoordinator

源码分析

以ActivityA -> ActivityB为例。

makeSceneTransitionAnimation

ActivityOptionsCompat.makeSceneTransitionAnimation方法内部主要负责：

  • 判断是否支持转场动画
  • 保存共享View信息
  • 生成ExitTransitionCoordinator，并保存到当前activity的mActivityTransitionState中

ExitTransitionCoordinator生成的时候,会对共享元素和非共享元素的进行区分，以及处理其他元素

退场动画的执行时机

调用startActivity会按以下顺序执行:
startActivity->startActivityForResult->cancelInputsAndStartExitTransition->startExitOutTransition->startExit

在调用startActivity启动新Activity之后，会立刻调用退场动画的执行

退场动画的流程

这里的moveSharedElementsToOverlay方法比较重要，会使用到最开始介绍的GhostView和OverlayView ，目的是将共享元素绘制到最顶层

beginTransitions方法里主要是利用TransitionManager.beginDelayedTransition执行过渡动画，需要注意的是，这里会将共享元素，与非共享元素的动画合并成一个Transition，同时在动画前后处理GhostView的可见状态，用于收集开始帧和结束帧的信息。
上面的sharedElementTransition和viewsTransition都添加了监听器，在动画结束之后分别调用sharedElementTransitionComplete和viewsTransitionComplete方法

然后在startInputWhenTransitionsComplete方法里会调用onTransitionsComplete方法，最终会调用notifyComplete方法通知退场动画已经完成。

进场动画的执行时机

在新的Activity中的performaStart中会创建EnterTransitionCoordinator

EnterTransitionCoordinator在创建的时候主要处理了以下事情：

  • 保存ExitTransitionCoordinator对象到mResultReceiver中
  • 将Activity的window背景设置成透明
  • 发送MSG_SET_REMOTE_RECEIVER消息，将EnterTransitionCoordinator对象传递给ActivityA

接着回到前面构建EnterTransitionCoordinator之后，如果没有设置延迟，就会直接调用startEnter()

根据上面的代码流程startEnter()方法最终是执行到EnterTransitionCoordinator.viewsReady。该方法里面也是主要对元素进行划分处理，与退场动画一致，但是首次执行到这里的时候
mSharedElementsBundle是为null的，所以不会走onTakeSharedElements方法,这里的mSharedElementsBundle对象是在ActivityA的notifyComplete中发送的MSG_TAKE_SHARED_ELEMENTS消息传过来的

可以看到，当收到退场动画完成的时候，会执行onTakeSharedElements方法。

最终会调用startSharedElementTransition方法

该方法是执行动画前最后的步骤，主要流程为：

  • 根据ActivityA传过来的共享元素sharedElementState信息，创建快照view对象
  • 针对非共享元素执行渐隐的退出动画
  • 创建共享元素的快照view,并显示共享元素
  • 设置共享元素设置到动画的开始位置，并返回在ActivityB布局中的原始的状态（即结束位置）
  • 使用TransitionManager，采集开始帧和结束帧，执行过度动画
  • 将共享元素设置到结束的位置（为了TransitionManager能采集到结束帧的值）

进场动画的流程

根据上面进场动画的执行时机，最终是会走到beginTransition

该方法与退场动画类似，都是合并共享元素与非共享元素的动画为一个Transition，然后执行动画效果。
当动画执行结束之后就会触发 onTransitionsComplete方法。

用非常简单点的话总结共享元素流程是：
ActivityA先执行退场动画
ActivityA将共享元素的结束位置等属性传递给ActivityB
ActivityB加载自己的布局，在onStart生命周期左右去找到共享元素 先定位到ActivityA的结束位置
ActivityB开始执行过度动画，过渡到自己布局中的位置

使用注意事项

特别需要注意的，退场动画会在生命周期onStop()的时候结束所有过渡动画

该resetViews()的方法，会在Activity.onStop方法中进行调用
