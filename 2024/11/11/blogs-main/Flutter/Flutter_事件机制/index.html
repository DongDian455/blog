触摸事件流程

触摸事件通常针对移动端的手势处理，而手势处理流程一般为：
寻找可以响应该事件的视图->该视图是否消费该事件

Flutter的触摸事件处理

Flutter的触摸事件处理一样符合传统的手势处理流程。只不过Flutter将手势流程分为三个步骤：

  1. 命中测试 : 即寻找那些视图可以响应该事件
  2. 事件分发 : 从找到的视图中进行事件分发
  3. 事件清理 : 从事件分发的列表中移除视图

一个对象是否可以响应事件，取决于在其对命中测试过程中是否被添加到了 HitTestResult 列表 ，如果没有被添加进去，则后续的事件分发将不会分发给自己。

单child和多child的命中测试处理

因为RenderView只有一个孩子，所以直接调用child.hitTest即可。
而如果存在child不止一个，那么就需要判断每一个child，看下他们是否可以命中测试。

单child

以RenderBox为例：

RenderBox首先会判断点击位置是否在自身的范围内，如果是则进行后续判断。

  • hitTestChildren : 主要用于测试child的命中逻辑
  • hitTestSelf : 用于判断自身是否响应命中测试

如果不重写hitTestChildren，则默认直接返回 false，这也就意味着后代节点将无法参与命中测试，相当于事件被拦截了，这也正是 IgnorePointer 和 AbsorbPointer 可以拦截事件下发的原理。
如果hitTestSelf返回 true，则无论子节点中是否有通过命中测试的节点，当前节点自身都会被添加到HitTestResult中。而IgnorePointer和AbsorbPointer 的区别就是，前者的hitTestSelf返回了 false，而后者返回了 true。

  • IgnorePointer:忽略自身以及子节点
  • AbsorbPointer:只忽略子节点，自身仍可以响应触摸事件

注意这里有个关键点，Flutter与原生Andoid,IOS不同，Flutter的事件处理设计为： 只要子view命中了测试，那么自身也会进行命中，所以子view拿到了事件处理，后续分发事件的时候，父view也可以拿到该事件，至于事件是否处理，就是手势管理的逻辑。

HitTestBehavior

从前面分析可以得出，hitTestChildren和hitTestSelf方法是影响命中测试的关键方法，而Flutter还提供了一种管理命中测试的方式，那就是HitTestBehavior,它实际是一个枚举类型：

HitTestBehavior是一种命中测试的代理管理方式，对原本的命中测试做了一层封装，增加了一些命中测试的条件，用于外层能够直接管理命中测试，而不是每次都自定义类来管理命中测试。

在Listener和GestureDetector组件中，都提供了HitTestBehavior的设置，其源码在RenderProxyBoxWithHitTestBehavior中体现：

HitTestBehavior分别加入到了hitTest和hitTestSelf当作，其值也会决定了视图的命中测试结果，三个类型的作用效果如下：

  • deferToChild : 组件是否通过命中测试取决于子组件是否通过命中测试
  • opaque : 组件必然会通过命中测试，同时其 hitTest 返回值始终为 true
  • translucent : 组件必然会通过命中测试，但其 hitTest 返回值可能为 true 也可能为 false

opaque和translucent都会将让当前的组件通过命中测试，只有当其子节点的 hitTest返回为false时才会有区别。

  • 🤔️：什么场景下子节点hitTest会返回false?
    通常为GestureDetector包括子节点的时候的空白区域，点击空白区域默人就会返回false。其实也就是点击范围不在响应区域内。


  • 🤔️：GestureDetector设置translucent和opaque后，为什么空白区域也能响应事件？
    因为设置之后GestureDetector本身一定会通过命中测试。


  • 🤔️：GestureDetector设置translucent和opaque区别是什么？
    translucent不会拦截空白区域的事件，事件继续传递，而opaque会将事件进行拦截，原因是设置opaque，hitTestSelf一定返回true，即hitTest也会返回true,，又由于在多child的场景下，有一个子child命中了，后续的child就不会遍历了，所以opaque能够对于空白区域做事件拦截

    （为什么是空白区域？因为非空白区域通常hitTestSelf就会返回true,hitTest也会返回true，那就没有HitTestBehavior的事情了）

事件分发

当命中测试处理完，收集完能够响应事件的视图之后，就是对事件进行分发，而分发的过程就是对HitTestResult进行遍历处理

在对应的组件中，只需要重写handleEvent方法就可以自行处理事件逻辑。但是需要注意的事，由于Flutter是一个跨平台的框架，所以其对于事件的设计是最原始数据，即handleEvent获取的信息PointerEvent是很基础的数据。

手势管理

在事件分发的时候可以看到，只要在命中测试的列表里面，都可以收到事件分发，
那么会有一个问题:

  • 🤔️ 父布局和子布局同时收到事件消息，同时处理的话不会存在问题吗?
    这里就引出一个Flutter和原生客户端（Android/IOS）对于事件分发的处理有着很大的区别。
    Flutter对于不会阻止事件冒泡，而原生客户端可以通过自定义的方式阻止事件冒泡。
    Flutter对于事件处理是通过GestureRecognizer来处理的。对于需要处理手势的组件，会通过GestureRecognizer的竞争管理来进行处理。

Flutter手势管理有几个核心成员，分别为：

  • GestureArenaMember : 手势竞争者，GestureRecognizer就是继承该类
  • GestureArenaManager : 手势竞技场，每个竞技场存在一个或多个手势竞争者
  • GestureBinding : 负责维护和创建手势竞技场。

GestureDetector

通常需要手势的时候都是直接使用GestureDetector,但它只是一个包装类，包装了RawGestureDetector，而RawGestureDetector本质也是通过Listener进行触摸事件监听。RawGestureDetector内部维护了一个_recognizers，即手势竞争者集合

当Listener组件监听到点击事件的时候会触发RawGestureDetector._handlePointerDown方法，内部会遍历各个手势竞争者，并调用其addPointer方法。

手势竞争

GestureRecognizer的handleEvent中会识别手势，如果发生了某个手势，竞争者可以宣布自己是否胜出，一旦有一个竞争者胜出，竞技场管理者（GestureArenaManager）就会通知其他竞争者失败。胜出者的 acceptGesture 会被调用，其余的 rejectGesture 将会被调用。
