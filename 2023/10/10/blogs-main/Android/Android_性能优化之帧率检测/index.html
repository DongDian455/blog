帧率检测方法

在Android中，帧率检测实际为检测每一次收到Vsync信号后，所执行的任务时间。常见的检测方式有多种：

  • GPU渲染分析模式
  • Choreographer
  • OnFrameMetricsAvailableListener (API 24+)

方式一: GPU渲染分析模式

该方式为Android系统自带的功能，通常在手机系统的开发模式中可以打开

横向的绿线代表16.67ms的基线，小于这个说明fps为标准范围内，同时也能打开另外一个“显示GPU视图更新”，打开后可以看到哪些视图正在刷新


该方法简单易用，不过一般只用于滑动或者动画的时候检测，因为他无法知道具体是哪些函数，而且线上无法使用，只能开发/测试调试用

方式二: Choreographer

通过前面Choreographer的内容可知，Choreographer能过监听Vsync信号的回调，通过检测两次回调的时间差可以知道当前帧率的情况.
该方式较为简单，但是存在以下问题：

  1. 每次使用Choreographer添加监听，实际是申请一次Vsync信号的回调，而且由于监听任务只有一次有效，所以需要一直循环监听。而当前可能不存在界面的刷新任务，所以会造成资源浪费。
  2. 每一帧的任务包含了Choreographer内部的5种链表数组任务（input,animation,inset_animation…），在做优化的时候，可能需要具体的任务耗时情况。如果只通过两帧之间的时间是无法进行区分。

优化思路一

即然通过Choreographer的回调方法设置监听会存在资源浪费，那么就不进行回调监听设置，而是通过Looper的方式来监听每一个Message的任务执行时间（因为Choreographer对于Vsync的信号回调执行也是通过在主线程的Handler进行发送任务，即其doFrame的任务都是在主线中执行，所以可以通过主线程的Looper来监听这次任务的耗时时间）
Looper与Choreographer的关系可以参考下图：

那么，如何监听Looper对于Message的执行的开始和结束时间？
可以采用的是Printer方案，这也是比较基础的 APM 方案，很多地方都有其身影。简而言之，就是给主线程 Looper 设置一个 Printer，根据源码可以得知，消息开始执行和结束前，都会使用 Printer 打印一段日志。通过我们制定的 Printer 来对打印的日志分析，可以得知目前的是消息的开始还是结束，从而完成对消息处理的监听。

那么现在的问题就变成，如何知道这条消息是属于Choreographer发送的绘制消息?

优化思路二

Choreographer在收到Vsync信号的时候，会对其内部的5种类型的事件链表数组进行按顺序处理，而这5种类型的任务加起来就是一帧所消耗的时间。如果可以在在方法的开始结束，以及每一个链表数组的开头插入hook点,那么就可以表示当前正在处理的消息是UI绘制消息，并且也能计算出具体每个事件任务的耗时情况。解决前面提到的第二个问题。大致流程图如下：

方法三：OnFrameMetricsAvailableListener (API 24+)

从 7.0（API 24）开始，安卓 SDK 新增OnFrameMetricsAvailableListener接口用于提供帧绘制各阶段的耗时，数据源与 GPU Profile 相同。

使用方式：

通常情况，可以在Application监听Activity的生命周期回调，在onStart的时候设置，在onStop的时候移除监听

FrameMetrics变量说明

参数常量 代表的含义
ANIMATION_DURATION 动画执行回调时间, 单位纳秒
COMMAND_ISSUE_DURATION 向 GPU 发送绘制命令花费的时间, 单位纳秒
DRAW_DURATION 将组件树 ( View Hierarchy ) 转为显示列表 ( DisplayLists ) , 计算过程所花费的时间, 单位纳秒
FIRST_DRAW_FRAME 绘制的该帧是否是第一帧, 0 是, 1 不是 ; 第一帧渲染会慢一些 , 第一帧不会引发动画中的跳帧问题, 这些问题都会被窗口动画隐藏 , 不必进行显示过程中的 jank 计算
INPUT_HANDLING_DURATION 处理输入事件花费的时间, 单位纳秒
INTENDED_VSYNC_TIMESTAMP 该值是个时间戳, 表示该帧的 vsync 信号发出时间 ; 这个时间是当前帧的预期开始时间 , 如果该时间与 VSYNC_TIMESTAMP 时间戳不同 , 那么说明 UI 线程被阻塞了, 没有及时响应 vsync 信号
LAYOUT_MEASURE_DURATION 组件树 ( view hierarchy ) 测量 ( measure ) 和摆放 ( layout ) 花费的时间 , 单位纳秒
SWAP_BUFFERS_DURATION CPU 传递多维向量图形数据给 GPU 花费的时间, 单位纳秒
SYNC_DURATION 显示列表 ( DisplayLists ) 与显示线程同步花费的时间, 单位纳秒
TOTAL_DURATION CPU 渲染到传递到 GPU 所用的总时间, 上述所花费的有意义的时间之和 , 单位纳秒
UNKNOWN_DELAY_DURATION UI 线程响应并开始处理渲染的等待时间, 一般是 0, 如果大于 0 说明出问题了
VSYNC_TIMESTAMP vsync 信号发出的时间戳, 该时刻 GPU 应该进行绘制, 间隔 16ms , 同时 CPU 开始渲染
